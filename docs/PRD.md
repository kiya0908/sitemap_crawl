# Sitemap Crawl 产品需求文档（PRD）

## 1. 文档信息

- 项目名称：Sitemap Crawl
- 产品形态：仅供个人使用的私有 SEO 管理后台
- 部署目标：Cloudflare
- 文档状态：MVP 已确认
- 默认业务时区：Asia/Tokyo

## 2. 产品背景

在竞品 SEO 研究中，人工定期检查 Sitemap、识别新增页面、整理页面标题和推测目标关键词，过程重复且容易遗漏。

本项目通过定时抓取竞品 Sitemap、保存 URL 历史并执行增量对比，自动发现竞品新上线或新加入 Sitemap 的页面；随后抓取新增页面的基础 SEO 信息，并通过 OpenRouter 输出结构化的页面类型、主题、推测关键词与搜索意图，供人工复核和后续 SEO 决策使用。

## 3. 产品目标

### 3.1 核心目标

1. 自动监控 3–5 个竞品网站的 Sitemap。
2. 第一次扫描建立基线，从第二次扫描开始识别新增页面。
3. 对新增页面抓取 SEO 元素并进行 AI 结构化分析。
4. 在私有后台中查看、筛选、人工复核、备注和导出结果。
5. 在不保存完整 HTML 和每日全量快照的前提下控制数据库增长。

### 3.2 非目标

MVP 不用于：

- 获取竞品真实排名关键词；
- 监控所有页面正文变化；
- 替代 Ahrefs、Semrush、GSC 或 SERP 排名工具；
- 自动生成或发布文章；
- 对外提供 SaaS 服务；
- 多用户、支付、套餐和权限管理。

## 4. 用户与使用场景

### 4.1 用户

仅项目所有者本人使用。后台通过 Cloudflare Access 保护，应用内部不开发注册、登录、找回密码和用户管理功能。

### 4.2 主要使用场景

1. 添加一个竞品域名和一个或多个 Sitemap 地址。
2. 首次执行扫描，建立现有 URL 基线。
3. 每天自动扫描，识别历史中从未出现的新 URL。
4. 查看新增页面对应的标题、H1、页面类型和推测关键词。
5. 将页面标记为“值得跟进”或“无需跟进”，并添加个人备注。
6. 按竞品、时间、页面类型、搜索意图和人工状态筛选结果。
7. 将筛选后的结果导出为 CSV，用于进一步分析。

## 5. 规模与性能边界

- 竞品数量：3–5 个。
- 单个竞品 URL：不超过 1,000 个。
- 总 URL：预计不超过 5,000 个。
- 默认自动扫描频率：每天一次。
- 支持后台手动立即扫描。
- MVP 不为大规模分布式抓取设计。

## 6. 核心业务流程

```text
添加竞品
  ↓
配置或自动识别 Sitemap
  ↓
首次扫描并建立 URL 基线
  ↓
每日 Cron / 手动触发扫描
  ↓
解析 Sitemap Index 与子 Sitemap
  ↓
URL 标准化并与历史记录对比
  ↓
识别新增、消失、重新出现的 URL
  ↓
只抓取新增或需要重试的页面
  ↓
提取基础 SEO 信息
  ↓
调用 OpenRouter 结构化分析
  ↓
后台查看、复核、备注和导出
```

## 7. 功能需求

## 7.1 Dashboard

展示以下汇总指标：

- 已启用竞品数量；
- 最近一次扫描状态和时间；
- 今日新增页面数；
- 最近 7 天新增页面数；
- 待人工查看页面数；
- 值得跟进页面数；
- 最近失败扫描或抓取数量。

MVP 以数字卡片和最近活动列表为主，不要求复杂图表。

## 7.2 竞品管理

每个竞品包含：

- 名称；
- 域名；
- 是否启用；
- 创建时间；
- 最后扫描时间；
- 最后扫描状态；
- 基线是否已建立。

支持：

- 新增竞品；
- 编辑竞品名称、域名和启用状态；
- 暂停或恢复监控；
- 手动触发扫描；
- 查看扫描历史；
- 删除竞品前二次确认。

删除策略：MVP 默认软删除竞品，保留历史数据，避免误删。

## 7.3 Sitemap 管理

每个竞品支持一个或多个 Sitemap。

支持两种来源：

1. 手动填写 Sitemap URL，优先级最高；
2. 自动识别作为辅助：
   - 读取 `robots.txt` 中的 Sitemap 声明；
   - 检查常见路径：`/sitemap.xml`、`/sitemap_index.xml`、`/wp-sitemap.xml`。

解析要求：

- 支持 Sitemap Index；
- 支持递归解析子 Sitemap；
- 防止循环引用；
- 对重复 URL 去重；
- 记录每个 URL 来自哪个 Sitemap；
- XML 解析失败时记录明确错误；
- 不长期保存完整 XML 内容。

MVP 不处理图片 Sitemap、视频 Sitemap 的媒体内容；其中的普通页面 URL 可按 HTML 页面处理。

## 7.4 首次基线扫描

首次成功扫描时：

- 将当前 Sitemap 中的 URL 写入页面主表；
- 页面系统来源状态标记为 `baseline`；
- 不创建“新增页面”通知或待处理记录；
- 不对全部基线页面调用 OpenRouter；
- 可选择只保存 Sitemap 字段，不抓取全部页面正文。

基线只有在一次完整成功的扫描后才标记为已建立。若 Sitemap 解析不完整或扫描失败，不得误将不完整结果作为基线。

## 7.5 增量 URL 对比

从第二次完整扫描开始执行对比。

### 新增页面

满足以下条件时标记为 `new`：

- 当前扫描中存在；
- 同一竞品历史记录中从未出现过对应的标准化 URL。

### 消失页面

- 第一次未出现在完整扫描结果中：增加连续缺失计数，不立即确认消失；
- 连续两次完整扫描未出现：标记为 `missing`；
- 扫描不完整或失败时不得增加缺失计数。

### 重新出现

历史页面已经标记为 `missing`，之后再次出现在完整扫描中时，标记为 `reappeared`，不作为首次新增。

## 7.6 URL 标准化

比较前必须生成 `normalized_url`。默认规则：

- 协议与域名转为小写；
- 移除 Fragment；
- 统一默认端口；
- 规范化重复斜杠；
- 统一尾斜杠策略；
- 移除常见追踪参数，如 `utm_*`、`gclid`、`fbclid`；
- 对剩余查询参数按键排序；
- 不武断移除可能影响页面内容的业务参数；
- 保留原始 URL 供展示和排错。

唯一约束为：`competitor_id + normalized_url`。

## 7.7 新增页面抓取

只对以下对象执行页面抓取：

- 新增页面；
- 重新出现且需要重新确认的页面；
- 之前抓取失败、符合重试条件的页面；
- 用户手动要求重新抓取的页面。

提取字段：

- HTTP 状态码；
- 最终 URL；
- Redirect 链摘要；
- Title；
- Meta Description；
- H1；
- H2 列表；
- Canonical；
- Robots Meta；
- 页面语言；
- 页面正文清洗摘要；
- 内容哈希；
- 抓取时间；
- 抓取状态和错误信息。

抓取限制：

- 只处理可识别的 HTML 页面；
- PDF 仅保留 URL 和响应信息，MVP 不解析正文；
- 不保存完整 HTML；
- 正文清洗摘要需设置最大字符数，建议 5,000–10,000 字符；
- 设置超时、User-Agent 和单站请求间隔；
- 尊重目标网站的稳定性，避免并发轰炸。

## 7.8 OpenRouter AI 分析

OpenRouter 必须封装为独立 Provider，模型通过环境变量配置，不得写死在业务代码中。

输入信息优先包括：

- URL 与 slug；
- Title；
- Meta Description；
- H1、H2；
- 清洗后的正文摘要；
- 页面语言。

输出必须为可验证的结构化 JSON：

```json
{
  "pageType": "product",
  "primaryTopic": "dynamic mixing nozzle",
  "primaryKeyword": "dynamic mixing nozzle for epoxy",
  "secondaryKeywords": [
    "epoxy mixing nozzle",
    "two component adhesive nozzle"
  ],
  "searchIntent": "commercial",
  "productLine": "mixing nozzles",
  "summary": "面向环氧树脂双组分混合应用的产品页面",
  "reasoningEvidence": [
    "Title 包含 dynamic mixing nozzle for epoxy",
    "H1 与产品应用场景一致"
  ],
  "confidence": 0.86
}
```

要求：

- 使用运行时 Schema 校验；
- 结构化输出解析失败时重试，最多 2 次；
- AI 失败不得丢失新增页面记录；
- 保存模型名称、分析时间、提示词版本和原始响应摘要；
- 明确标注关键词为“推测关键词”，不是竞品真实排名词；
- 人工修改值与 AI 原始值分开保存。

## 7.9 新增页面列表

表格字段：

- 首次发现时间；
- 竞品；
- 页面标题；
- URL；
- 页面类型；
- 推测主关键词；
- 搜索意图；
- 抓取状态；
- AI 分析状态；
- 人工复核状态；
- 是否值得跟进。

筛选条件：

- 竞品；
- 首次发现时间范围；
- 页面类型；
- 搜索意图；
- 抓取状态；
- 分析状态；
- 已查看 / 未查看；
- 值得跟进 / 无需跟进；
- URL、标题或关键词搜索。

支持分页、排序和 CSV 导出。导出内容应遵循当前筛选条件。

## 7.10 页面详情与人工复核

展示：

- 原始 URL 与标准化 URL；
- 首次发现和最后出现时间；
- Sitemap 来源与 `lastmod`；
- HTTP、Redirect、Canonical 和 Robots 信息；
- Title、Meta、H1、H2；
- 页面正文摘要；
- AI 分析结果和判断依据；
- AI 模型与分析时间；
- 抓取及分析错误；
- 页面事件时间线。

支持人工操作：

- 标记已查看；
- 标记值得跟进；
- 标记无需跟进；
- 编辑人工主关键词、次级关键词、页面类型和搜索意图；
- 添加个人备注；
- 手动重新抓取；
- 手动重新分析。

## 7.11 扫描记录

每次扫描保存：

- 竞品；
- 触发方式：Cron / 手动；
- 开始与结束时间；
- 扫描状态；
- Sitemap 数量；
- URL 总数；
- 新增、缺失、重新出现数量；
- 抓取成功和失败数量；
- AI 分析成功和失败数量；
- 错误摘要。

扫描状态建议：

- `queued`
- `running`
- `success`
- `partial_success`
- `failed`

只有 `success` 或确认完整的 `partial_success` 才能参与缺失页面判断。

## 8. 状态模型

### 8.1 页面 Sitemap 生命周期状态

- `baseline`：首次扫描建立的存量页面；
- `active`：当前仍存在于 Sitemap；
- `new`：本次首次发现；
- `missing`：连续两次完整扫描未出现；
- `reappeared`：消失后重新出现。

实现时可将“来源事件”和“当前生命周期状态”分开存储，避免一个字段承担多个含义。

### 8.2 抓取状态

- `pending`
- `fetching`
- `success`
- `failed`
- `unsupported`

### 8.3 AI 分析状态

- `pending`
- `analyzing`
- `success`
- `failed`
- `skipped`

### 8.4 人工复核状态

- `unreviewed`
- `reviewed`
- `worth_following`
- `not_relevant`

## 9. 数据保留策略

长期保存：

- 竞品和 Sitemap 配置；
- 页面主记录；
- AI 分析结果；
- 人工修改和备注；
- 新增、消失、重新出现事件；
- 扫描汇总记录。

限制或不保存：

- 不保存完整 Sitemap XML；
- 不保存完整 HTML；
- 不保存每日全量 URL 快照；
- 详细失败日志默认保留 90 天；
- 页面正文摘要设最大长度；
- AI 原始响应设最大长度并避免存储敏感信息。

## 10. 安全与配置

- 后台由 Cloudflare Access 保护；
- 应用不实现用户认证；
- `OPENROUTER_API_KEY` 使用 Cloudflare Secret；
- 所有密钥不得出现在仓库、日志和前端响应中；
- 手动扫描接口必须处于 Access 保护范围内，并具备服务端校验；
- Cron 入口不得依赖浏览器端认证；
- 对外抓取 URL 必须防止 SSRF，限制为已配置竞品域名及其允许的 Sitemap 主机；
- 公共仓库不得提交真实域名列表、私密备注或生产数据库内容。

## 11. MVP 验收标准

1. 可新增、编辑、暂停 3–5 个竞品。
2. 每个竞品可配置多个 Sitemap，并能解析 Sitemap Index。
3. 首次完整扫描只建立基线，不产生新增页面。
4. 第二次扫描能准确识别新 URL。
5. 同一 URL 不会因追踪参数或尾斜杠差异重复创建。
6. 页面连续两次缺失后标记为 `missing`。
7. 新增页面可抓取 Title、Meta、H1、H2、Canonical 和正文摘要。
8. OpenRouter 输出通过 Schema 校验后保存；失败可重试且不影响新增记录。
9. 后台可筛选、复核、备注并导出 CSV。
10. Cron 每天运行一次，手动扫描可用。
11. 完整 HTML、完整 Sitemap 和每日全量快照未写入 D1。
12. 所有密钥均通过环境变量或 Secret 管理。

## 12. 后续版本候选项

- 邮件或其他渠道的每日/每周摘要；
- 页面内容变化监控；
- 自有网站 Sitemap 与竞品内容缺口对比；
- 主题聚类和竞品布局趋势；
- SERP 或第三方关键词数据验证；
- 自动生成建议页面 Brief；
- 更灵活的扫描频率；
- R2 保存临时原始响应以便排错。
